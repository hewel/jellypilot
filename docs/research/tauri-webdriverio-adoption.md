# Tauri WebdriverIO adoption for JellyPilot

## Decision

JellyPilot should continue toward a Linux-only embedded-WebDriver harness, but it should not implement the downloaded guide verbatim. The upstream architecture is valid: WebdriverIO 9 drives a compiled Tauri 2 application through `@wdio/tauri-service`; `tauri-plugin-wdio-webdriver` hosts the embedded WebDriver server; and `tauri-plugin-wdio` plus its frontend bridge provide Tauri execution, command mocking, and log forwarding. Current first-party releases are mutually compatible with JellyPilot's declared Tauri, Rust, and Node floors. ([WebdriverIO plugin setup](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/), [npm service metadata](https://registry.npmjs.org/@wdio/tauri-service/latest), [WDIO crate](https://crates.io/crates/tauri-plugin-wdio), [embedded-driver crate](https://crates.io/crates/tauri-plugin-wdio-webdriver))

Adopt these boundaries:

- keep both Rust plugins optional behind one test-only Cargo feature;
- use a distinct Tauri configuration/capability and explicitly select production capabilities;
- use a test-only frontend entry that imports `@wdio/tauri-plugin` before application mount;
- retain Rsbuild, Rstest, Panda, TypeScript 7/TSGo, and Bun in their current roles; and
- require a native Linux/Xvfb prototype before claiming that launch, startup mocking, log capture, or teardown works.

This research authorizes no dependency, configuration, application, or workflow changes. The exact test build and its production-exclusion checks remain a decision for [Choose the E2E build flavor and production-isolation contract](https://github.com/hewel/jellypilot/issues/152); native launch evidence belongs to [Prove an isolated embedded-WebDriver launch on current main](https://github.com/hewel/jellypilot/issues/151); deterministic pre-mount command control belongs to [Prove deterministic Tauri command control before application mount](https://github.com/hewel/jellypilot/issues/155).

## Guide verdicts

| Guide claim | Verdict | JellyPilot adaptation |
|---|---|---|
| Use WebdriverIO for compiled-desktop workflows and keep frontend/Rust tests in their existing runners. | **Retain** | Replace the guide's Vitest wording with Rstest. Rstest continues to own jsdom unit/component tests; WDIO owns only native desktop E2E behavior. ([Rstest introduction](https://rstest.rs/guide/start/)) |
| Use `@wdio/tauri-service` with `driverProvider: "embedded"` and both Rust plugins. | **Retain** | The embedded plugin removes the separate `tauri-driver`/WebKitWebDriver requirement, but the Linux WebView still needs a display. Explicit provider selection is appropriate on Linux. ([plugin setup](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/), [platform support](https://webdriver.io/docs/desktop-testing/tauri/platform-support/)) |
| Add the Rust plugins as optional dependencies behind a feature. | **Retain** | A feature is preferable to `debug_assertions`: it supports a deliberate debug or optimized test build while keeping normal builds dependency-free. The exact feature name, release policy, and negative compile checks belong to [Choose the E2E build flavor and production-isolation contract](https://github.com/hewel/jellypilot/issues/152). ([plugin production guidance](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#production-considerations)) |
| Put `wdio:default` and `wdio-webdriver:default` in a dedicated capability, and explicitly select production capabilities. | **Retain** | JellyPilot currently has `capabilities/default.json` but does not explicitly select it in `tauri.conf.json`. Tauri enables every capability file by default until an explicit list is supplied, so merely adding an E2E file would expose it to normal builds. `wdio-webdriver:default` currently grants no IPC commands but should remain declared so its ACL manifest is loaded. ([Tauri capabilities](https://v2.tauri.app/security/capabilities/), [embedded plugin permissions](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#plugin-2-tauri-plugin-wdio-webdriver-optional---embedded-webdriver)) |
| Enable `withGlobalTauri` for the test build. | **Retain, test-only** | Keep it in the E2E overlay and prove the production config remains false/absent. The WDIO setup requires the global API. ([plugin setup](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/)) |
| Conditionally import the frontend bridge with `VITE_E2E`. | **Adapt** | JellyPilot uses Rsbuild, whose default public prefix is `PUBLIC_`; use a test-only entry selected by `PUBLIC_WEBDRIVER`, or an equivalent explicit Rsbuild definition. Do not add Vite conventions. The bridge must load before mount because it installs `window.wdioTauri`, console forwarding, and `invoke` interception. ([Rsbuild public variables](https://rsbuild.rs/guide/advanced/env-vars#public-variables), [plugin lifecycle](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#plugin-lifecycle)) |
| Mock commands with `browser.tauri.mock()` and restore them after each test. | **Retain with a limit** | WDIO implements mocks in the frontend by intercepting `window.__TAURI__.core.invoke`; no Rust mock permissions exist. A mock must be registered before its command is invoked. JellyPilot mounts immediately and performs startup work, so post-session mocks cannot yet guarantee control of startup commands. [Prove deterministic Tauri command control before application mount](https://github.com/hewel/jellypilot/issues/155) must prove a deferred-mount entry/alias or another fail-closed seam, plus one safe real IPC command. ([mocking setup](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#mocking-doesnt-work), [usage examples](https://webdriver.io/docs/desktop-testing/tauri/usage-examples/)) |
| Register WDIO plugins in the Tauri builder and enable frontend/backend capture. | **Adapt and defer exact order** | Upstream says the WDIO plugin may appear anywhere, typically after existing plugins. Backend capture reads process stdout; frontend capture polls WebDriver logs and can silently be unavailable on some drivers. JellyPilot currently installs `tauri-plugin-log` during `setup`, so [Choose the E2E build flavor and production-isolation contract](https://github.com/hewel/jellypilot/issues/152) must choose the builder/logging order and [Prove an isolated embedded-WebDriver launch on current main](https://github.com/hewel/jellypilot/issues/151) must prove both streams on Linux rather than assuming them. ([plugin registration](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#step-2-register-plugin-in-rust), [log forwarding](https://webdriver.io/docs/desktop-testing/tauri/log-forwarding/)) |
| Use Xvfb in headless Linux CI. | **Retain** | The v1 target is Linux only. Use `xvfb-run -a` or the service's documented auto-detection, then prove WebKit/Tauri startup under the actual runner in [Prove an isolated embedded-WebDriver launch on current main](https://github.com/hewel/jellypilot/issues/151). Do not copy the guide's Windows/macOS matrix into v1. ([Linux headless testing](https://webdriver.io/docs/desktop-testing/tauri/platform-support/#headless-testing)) |
| Let the service terminate the application and retain screenshots/logs on failure. | **Retain as an acceptance target, defer proof** | Upstream documents service-owned process termination and optional log forwarding, but JellyPilot still needs repeated clean-exit, no-orphan, screenshot, and log evidence under Linux in [Prove an isolated embedded-WebDriver launch on current main](https://github.com/hewel/jellypilot/issues/151). ([embedded-provider lifecycle](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#how-it-works-1), [log forwarding](https://webdriver.io/docs/desktop-testing/tauri/log-forwarding/)) |
| Install `typescript`, use a standalone generic `tsconfig.e2e.json`, and rely on WDIO to compile it. | **Reject as written** | TypeScript 7 and Effect TSGo are already pinned. WDIO additionally needs `tsx`; transpilation does not type-check. The implementation should extend JellyPilot's TS configuration, add `node`, WDIO, and Mocha types, point WDIO at that config, and run an explicit TSGo typecheck. ([WDIO TypeScript setup](https://webdriver.io/docs/typescript/)) |
| Use npm/cross-env scripts that call Tauri and WDIO. | **Reject as written** | Use Bun scripts and the existing lockfile. On Linux, shell environment assignment is sufficient; adding `cross-env` is unnecessary. Preserve `bun tauri build` and the existing `beforeBuildCommand -> bun run build -> dist` path unless [Choose the E2E build flavor and production-isolation contract](https://github.com/hewel/jellypilot/issues/152) deliberately chooses another build seam. Whether the WDIO CLI itself runs under Bun or Node must be proven in [Prove an isolated embedded-WebDriver launch on current main](https://github.com/hewel/jellypilot/issues/151). ([Bun scripts](https://bun.sh/docs/runtime), [Tauri CLI](https://v2.tauri.app/reference/cli/)) |
| Treat E2E build/test commands as independent of Panda. | **Adapt** | JellyPilot's `prebuild` already runs `panda codegen --clean`, and production/E2E frontend compilation imports generated `@styled-system` output. The E2E path must preserve that lifecycle rather than add a competing watcher or commit generated output. ([Panda codegen](https://panda-css.com/docs/references/cli/#codegen), [styled system](https://panda-css.com/docs/concepts/styled-system)) |
| Start with cross-platform, persistence, and platform-feature suites. | **Reject for v1** | The map is Linux-only and intentionally excludes live servers, external MPV, playback, full restart persistence, Windows, and macOS. Initial coverage is startup, bridge readiness, one UI interaction, one safe real command, one mocked failure, teardown, evidence, and production isolation. |

## Local evidence and historical boundary

Current repository facts:

- `src/index.tsx` imports no WDIO code and mounts `<App />` immediately.
- `src-tauri/tauri.conf.json` has no explicit capability list and leaves `withGlobalTauri` absent; `capabilities/default.json` is the only current capability.
- the Rust logger is installed inside `setup` with stdout and WebView targets;
- `prebuild`, `pretest`, and `prepare` regenerate Panda output; Rstest has its own Solid/jsdom configuration; and
- TypeScript is exact-pinned at 7.0.2 and patched by Effect TSGo during `prepare`.

Retired commit [`738e2c8`](https://github.com/hewel/jellypilot/commit/738e2c8dd8afeb968a4aba169acddc4bcfb611c9) is useful historical evidence, not native-runtime proof. It established several reusable seams: exact WDIO/Rust package versions, an optional `webdriver` Cargo feature, `PUBLIC_WEBDRIVER`, a distinct Tauri identity, a bridge-first deferred-mount entry, a fail-closed alias for Tauri `invoke`, embedded-provider configuration, and frontend/backend capture options. Its commit message explicitly says native execution remained manual; the acceptance notes on [Create secure catalog and app WebView test seams](https://github.com/hewel/jellypilot/issues/107#issuecomment-4936471336) and [Validate the migrated app with deterministic IPC](https://github.com/hewel/jellypilot/issues/129#issuecomment-4936476962) left the native run unverified. Both issues were later closed as part of the abandoned UI Core/Atomic CSS direction. Therefore the commit proves that a scaffold was designed and build-shaped, not that a Tauri/WebKit process launched, the bridge initialized, mocks controlled startup, logs arrived, or teardown was clean.

Do not revive its UI Core entry, Atomic CSS plugin, broad fixture suite, or branch architecture. Re-evaluate only the narrow seams above against current Panda/main.

## Runtime-only unknowns and ownership

No new ticket is needed:

- [Prove an isolated embedded-WebDriver launch on current main](https://github.com/hewel/jellypilot/issues/151) owns Linux/Xvfb launch, bridge readiness, port behavior, Bun-versus-Node runner behavior, frontend/backend log capture, screenshots, isolated application/storage state, repeatability, and clean teardown.
- [Prove deterministic Tauri command control before application mount](https://github.com/hewel/jellypilot/issues/155) owns the bridge-first/deferred-mount seam, exact generated startup command names and wire shapes, fail-closed undeclared commands, restoration, and coexistence with one safe real IPC command.
- [Choose the E2E build flavor and production-isolation contract](https://github.com/hewel/jellypilot/issues/152) owns feature and overlay names, explicit production capability selection, identity/storage isolation, debug versus optimized binary policy, Rust plugin/logging order, frontend-entry selection, Panda/TSGo check integration, and negative checks proving WDIO code, globals, permissions, and dependencies do not enter shipped artifacts.

Until those tickets produce evidence, the supported conclusion is compatibility of the documented integration seams—not a claim that JellyPilot has a working or production-safe native E2E harness.
