# Tauri WebdriverIO adoption for JellyPilot

## Adoption decision and final evidence

JellyPilot should continue toward a Linux-only embedded-WebDriver harness, but it should not implement the downloaded guide verbatim. The upstream architecture is valid: WebdriverIO 9 drives a compiled Tauri 2 application through `@wdio/tauri-service`; `tauri-plugin-wdio-webdriver` hosts the embedded WebDriver server; and `tauri-plugin-wdio` plus its frontend bridge provide Tauri execution, command mocking, and log forwarding. Current first-party releases are mutually compatible with JellyPilot's declared Tauri, Rust, and Node floors. ([WebdriverIO plugin setup](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/), [npm service metadata](https://registry.npmjs.org/@wdio/tauri-service/latest), [WDIO crate](https://crates.io/crates/tauri-plugin-wdio), [embedded-driver crate](https://crates.io/crates/tauri-plugin-wdio-webdriver))

Adopt these boundaries:

- keep both Rust plugins optional behind one test-only Cargo feature;
- use a distinct Tauri configuration/capability and explicitly select production capabilities;
- use a test-only frontend entry that imports `@wdio/tauri-plugin` before application mount;
- retain Rsbuild, Rstest, Panda, TypeScript 7/TSGo, and Bun in their current roles; and
- require native Linux/Xvfb evidence for launch, startup command control, log capture, and teardown rather than inferring those behaviors from configuration or compilation.

The completed [Wayfinder map](https://github.com/hewel/jellypilot/issues/149) subsequently resolved the runtime and safety questions identified by this research. A disposable prototype launched the compiled Tauri/WebKitGTK application three times under isolated Linux/Xvfb sessions, initialized the bridge, captured frontend and backend logs, saved screenshots, and left no application, driver, Xvfb process, port, or production-storage change behind. It also proved that startup commands require a WebDriver-only entry, an exact `@tauri-apps/api/core` alias, and one-shot deferred mount; global WebdriverIO mocks alone do not intercept generated binding imports.

The accepted implementation uses an unbundled debug application behind the optional `webdriver` Cargo feature, rejects that feature when debug assertions are disabled, pins `@wdio/native-utils` to 2.5.0, installs the logger before the WebDriver plugins in feature builds, and keeps the existing setup-time logger behavior when the feature is off. A test-only Tauri overlay and capability use the isolated identifier `top.pigfun.jellypilot.webdriver`; normal production configuration explicitly selects only the default capability and must pass configuration, dependency-graph, frontend-output, binary-marker, and negative compile checks. The permanent harness itself remains unimplemented; this document records the evidence and decisions that authorize that implementation.

## Guide verdicts

| Guide claim | Verdict | JellyPilot adaptation |
|---|---|---|
| Use WebdriverIO for compiled-desktop workflows and keep frontend/Rust tests in their existing runners. | **Retain** | Replace the guide's Vitest wording with Rstest. Rstest continues to own jsdom unit/component tests; WDIO owns only native desktop E2E behavior. ([Rstest introduction](https://rstest.rs/guide/start/)) |
| Use `@wdio/tauri-service` with `driverProvider: "embedded"` and both Rust plugins. | **Retain** | The embedded plugin removes the separate `tauri-driver`/WebKitWebDriver requirement, but the Linux WebView still needs a display. Explicit provider selection is appropriate on Linux. ([plugin setup](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/), [platform support](https://webdriver.io/docs/desktop-testing/tauri/platform-support/)) |
| Add the Rust plugins as optional dependencies behind a feature. | **Retain** | Use the optional `webdriver` feature for an unbundled debug application while keeping normal builds dependency-free. Reject the feature when debug assertions are disabled, and verify the rejection through the production-isolation checks. ([plugin production guidance](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#production-considerations), [build decision](https://github.com/hewel/jellypilot/issues/152)) |
| Put `wdio:default` and `wdio-webdriver:default` in a dedicated capability, and explicitly select production capabilities. | **Retain** | JellyPilot currently has `capabilities/default.json` but does not explicitly select it in `tauri.conf.json`. Tauri enables every capability file by default until an explicit list is supplied, so merely adding an E2E file would expose it to normal builds. `wdio-webdriver:default` currently grants no IPC commands but should remain declared so its ACL manifest is loaded. ([Tauri capabilities](https://v2.tauri.app/security/capabilities/), [embedded plugin permissions](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#plugin-2-tauri-plugin-wdio-webdriver-optional---embedded-webdriver)) |
| Enable `withGlobalTauri` for the test build. | **Retain, test-only** | Keep it in the E2E overlay and prove the production config remains false/absent. The WDIO setup requires the global API. ([plugin setup](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/)) |
| Conditionally import the frontend bridge with `VITE_E2E`. | **Adapt** | JellyPilot uses Rsbuild, whose default public prefix is `PUBLIC_`; use a test-only entry selected by `PUBLIC_WEBDRIVER`, or an equivalent explicit Rsbuild definition. Do not add Vite conventions. The bridge must load before mount because it installs `window.wdioTauri`, console forwarding, and `invoke` interception. ([Rsbuild public variables](https://rsbuild.rs/guide/advanced/env-vars#public-variables), [plugin lifecycle](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#plugin-lifecycle)) |
| Mock commands with `browser.tauri.mock()` and restore them after each test. | **Retain with a limit** | WDIO implements mocks in the frontend by intercepting `window.__TAURI__.core.invoke`; no Rust mock permissions exist. The accepted startup seam therefore installs fixtures before mount through a dedicated entry and exact `@tauri-apps/api/core` alias, fails undeclared commands closed, and allows only centrally reviewed safe real commands, initially `config_default`. ([mocking setup](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#mocking-doesnt-work), [usage examples](https://webdriver.io/docs/desktop-testing/tauri/usage-examples/), [command-control proof](https://github.com/hewel/jellypilot/issues/155)) |
| Register WDIO plugins in the Tauri builder and enable frontend/backend capture. | **Adapt** | Backend capture reads process stdout; frontend capture polls WebDriver logs. The Linux prototype proved both streams and established that feature builds must install `tauri-plugin-log` before the WDIO plugins to avoid global logger initialization conflicts. Feature-off builds retain the current setup-time logger behavior. ([plugin registration](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#step-2-register-plugin-in-rust), [log forwarding](https://webdriver.io/docs/desktop-testing/tauri/log-forwarding/), [native proof](https://github.com/hewel/jellypilot/issues/151)) |
| Use Xvfb in headless Linux CI. | **Retain, local-first** | The v1 target is Linux-only local developer and agent verification, not CI. The prototype proved WebKit/Tauri startup with a private Xvfb display; the permanent harness must allocate a private auto-numbered display by default and offer explicit headed execution. Do not copy the guide's Windows/macOS matrix into v1. ([Linux headless testing](https://webdriver.io/docs/desktop-testing/tauri/platform-support/#headless-testing), [workflow decision](https://github.com/hewel/jellypilot/issues/150)) |
| Let the service terminate the application and retain screenshots/logs on failure. | **Retain** | Three native sessions proved service-owned clean termination, no orphan processes or ports, screenshots, and frontend/backend capture. The permanent workflow retains scrubbed failure-only evidence, verifies teardown diagnostics, and deletes only run-owned sandboxes. ([embedded-provider lifecycle](https://webdriver.io/docs/desktop-testing/tauri/plugin-setup/#how-it-works-1), [log forwarding](https://webdriver.io/docs/desktop-testing/tauri/log-forwarding/), [native proof](https://github.com/hewel/jellypilot/issues/151)) |
| Install `typescript`, use a standalone generic `tsconfig.e2e.json`, and rely on WDIO to compile it. | **Reject as written** | TypeScript 7 and Effect TSGo are already pinned. WDIO additionally needs `tsx`; transpilation does not type-check. The implementation should extend JellyPilot's TS configuration, add `node`, WDIO, and Mocha types, point WDIO at that config, and run an explicit TSGo typecheck. ([WDIO TypeScript setup](https://webdriver.io/docs/typescript/)) |
| Use npm/cross-env scripts that call Tauri and WDIO. | **Reject as written** | Use pinned Bun for repository scripts and the existing lockfile, with Node 22.11 or newer executing WDIO. The prototype proved Bun 1.3.14 orchestration with the WDIO CLI under Node 24.13.1. On Linux, shell environment assignment is sufficient; adding `cross-env` is unnecessary. ([Bun scripts](https://bun.sh/docs/runtime), [Tauri CLI](https://v2.tauri.app/reference/cli/), [native proof](https://github.com/hewel/jellypilot/issues/151)) |
| Treat E2E build/test commands as independent of Panda. | **Adapt** | JellyPilot's `prebuild` already runs `panda codegen --clean`, and production/E2E frontend compilation imports generated `@styled-system` output. The E2E path must preserve that lifecycle rather than add a competing watcher or commit generated output. ([Panda codegen](https://panda-css.com/docs/references/cli/#codegen), [styled system](https://panda-css.com/docs/concepts/styled-system)) |
| Start with cross-platform, persistence, and platform-feature suites. | **Reject for v1** | The map is Linux-only and intentionally excludes live servers, external MPV, playback, full restart persistence, Windows, and macOS. Initial coverage is startup, bridge readiness, one UI interaction, one safe real command, one mocked failure, teardown, evidence, and production isolation. |

## Local evidence and historical boundary

Current repository facts:

- `src/index.tsx` imports no WDIO code and mounts `<App />` immediately.
- `src-tauri/tauri.conf.json` has no explicit capability list and leaves `withGlobalTauri` absent; `capabilities/default.json` is the only current capability.
- the Rust logger is installed inside `setup` with stdout and WebView targets;
- `prebuild`, `pretest`, and `prepare` regenerate Panda output; Rstest has its own Solid/jsdom configuration; and
- TypeScript is exact-pinned at 7.0.2 and patched by Effect TSGo during `prepare`.

Retired commit [`738e2c8`](https://github.com/hewel/jellypilot/commit/738e2c8dd8afeb968a4aba169acddc4bcfb611c9) is useful historical evidence, not native-runtime proof. It established several reusable seams: exact WDIO/Rust package versions, an optional `webdriver` Cargo feature, `PUBLIC_WEBDRIVER`, a distinct Tauri identity, a bridge-first deferred-mount entry, a fail-closed alias for Tauri `invoke`, embedded-provider configuration, and frontend/backend capture options. Its commit message explicitly says native execution remained manual; the acceptance notes on [Create secure catalog and app WebView test seams](https://github.com/hewel/jellypilot/issues/107#issuecomment-4936471336) and [Validate the migrated app with deterministic IPC](https://github.com/hewel/jellypilot/issues/129#issuecomment-4936476962) left the native run unverified. Both issues were later closed as part of the abandoned UI Core/Atomic CSS direction. Therefore the commit proves that a scaffold was designed and build-shaped, not that a Tauri/WebKit process launched, the bridge initialized, mocks controlled startup, logs arrived, or teardown was clean.

Do not revive its UI Core entry, Atomic CSS plugin, broad fixture suite, or branch architecture. Re-evaluate only the narrow seams above against current Panda/main.

## Resolved implementation contract

The closed Wayfinder decisions now define an implementation-ready local Linux harness:

- [Native launch proof](https://github.com/hewel/jellypilot/issues/151) established bridge readiness, Bun/Node execution, frontend/backend capture, screenshots, isolated storage, three repeatable WebKitGTK sessions, and clean teardown.
- [Command-control proof](https://github.com/hewel/jellypilot/issues/155) established the bridge-first deferred entry, exact core alias, raw command wire behavior, fail-closed undeclared IPC, and coexistence with real `config_default`.
- [Build and isolation](https://github.com/hewel/jellypilot/issues/152) fixed the `webdriver` feature, test overlay and capability, isolated identity, debug-only policy, logger/plugin order, and executable production-exclusion checks.
- [Safe scenarios](https://github.com/hewel/jellypilot/issues/153) fixed the empty Login baseline, semantic selectors, real default configuration, typed password-connection failure, and explicit live-service, credential, playback, persistence, and cross-platform exclusions.
- [Local workflow](https://github.com/hewel/jellypilot/issues/150) fixed the incremental command surface, per-spec isolation, safe-real-command policy, timeouts, zero retries, retained scrubbed evidence, and agent verification policy without GitHub Actions or a PR gate.

These results prove feasibility and define the required safety boundaries. They do not mean the permanent harness exists yet; production implementation follows as a separate issue chain linked from [the completed map](https://github.com/hewel/jellypilot/issues/149).
